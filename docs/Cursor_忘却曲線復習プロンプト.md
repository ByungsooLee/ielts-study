# Cursor 実装プロンプト：忘却曲線に沿った自動復習（間隔反復・SM-2）

> 既存アプリに、**忘却曲線に基づく自動復習スケジューラ**を統合する指示。作り直さない・既存機能を壊さない。まず現状把握→計画→承認→実装。

---

## 背景（なぜこの設計か）
- **エビングハウスの忘却曲線**：記憶は時間とともに指数的に減衰する。だが**想起（思い出す）成功のたびに減衰が緩やかになり**、最適な間隔を空けて復習すると長期記憶に定着する（**分散効果**・**テスト効果**）。
- よって「**思い出せた度合いに応じて、次の復習間隔を伸ばす**」のが正解。間隔を広げる実証済みアルゴリズムの定番が **SM-2（SuperMemo 2、Anki等の基盤）**。本機能はこれを採用する。
- アプリの役割：毎日アプリを開くと、**忘れかけのタイミングの項目だけ**が自動でキューに並ぶ。新規は1日の上限内で `n`（通し番号）順に投入する。

## 前提データ
取り込みJSONの各 `StudyItem` は **`n`（通し番号・全教材で一意・不変）** を持つ。新規項目の投入順はこの `n` 昇順を使う。`type`（word/phrase/grammar/conversation）, `theme` 等は既存通り。

## スケジューリング状態（同期対象。既存のSRS状態があれば拡張/移行）
```ts
interface Sched {
  ef: number;        // ease factor 初期2.5, 下限1.3
  reps: number;      // 連続正解回数 初期0
  interval: number;  // 次回までの日数 初期0
  due: number;       // 復習予定日（通し日数 = Math.floor(Date.now()/86400000)）
  lapses: number;    // 失敗累計
  last: number;      // 最終学習 epoch(ms)（同期マージ用のtsを兼ねる）
  status: "new" | "learning" | "review" | "suspended";
}
// ProgressData.srs: Record<itemId, Sched> として保持し、KV同期。マージは last(ts) が新しい方を採用。
```
**移行**：既存が `{box,due,lapses,ts}` なら、`interval≈INTERVALS[box]`、`reps≈box`、`ef=2.5`、`last=ts` として一度だけ変換。未学習は `status:"new"`。

## アルゴリズム（SM-2）
採点は3ボタン。品質 `q` に対応（合格は `q>=3`）：
- **忘れた** → `q=1`（不合格）
- **あいまい**（思い出せたが苦労）→ `q=3`
- **覚えてた** → `q=5`

更新ロジック（`today=Math.floor(Date.now()/86400000)`）：
```
if (q < 3) {                      // 不合格＝再学習
  reps = 0; lapses += 1;
  interval = 0;                   // 当日キューに戻す（数枚あとに再出題）
  due = today;                    // ＝今日もう一度
  status = "learning";
} else {                          // 合格
  if (reps === 0)      interval = 1;       // 初回正解→翌日
  else if (reps === 1) interval = 6;       // 2回目→6日後
  else                 interval = Math.round(interval * ef);
  reps += 1;
  due = today + interval;
  status = "review";
}
// ease factor 更新（合否問わず）
ef = ef + (0.1 - (5 - q) * (0.08 + (5 - q) * 0.02));
if (ef < 1.3) ef = 1.3;
last = Date.now();
```
- 効果：覚えてた(q5)はEF微増で間隔が伸び、あいまい(q3)はEF微減で間隔が縮む。忘却曲線の「忘れる直前」に当たるよう自然に最適化される。
- 数値（初回1日・2回目6日・EF2.5・下限1.3）は定数として一箇所にまとめ、後で調整可能にする。

## 毎日の自動キュー（「今日の復習」）
アプリ起動時に自動生成：
1. **復習(due)**：`status∈{review,learning}` かつ `due <= today` を、`due` の古い順（より忘れている順）に。
2. **新規投入**：`status==="new"` を `n` 昇順で、**1日の新規上限**まで追加（既定 10、設定で 5/10/20 等）。1日にその日投入した新規数をカウントしてリセット。
3. 任意：当日の合計枚数に上限（例 200）。超過分は翌日へ。
- 「今日の復習 ○件（復習○＋新規○）」をホームに表示。終わると達成画面＋ストリーク。
- 既存の「セット学習(10/30/50・テーマ別)」とは**別モード**（自由練習）。セット学習で採点しても上記Schedは更新してよい（任意設定）。

## 例外処理
- **再学習(learning)**：不合格の項目は当日キュー末尾に戻し、その日に1〜2回再提示してから締める。
- **苦手/リーチ判定**：`lapses>=2` で苦手フラグ（既存の苦手フィルタに連動）。`lapses>=8` は「リーチ」として自動 `suspended` か警告（要復習として隔離）。
- **遅延した日**：何日か開かなかった場合、溜まったdueをまとめて出す（順序は due 古い順）。1日上限で分割表示してもよい。

## 可視化（任意だが推奨）
- 各項目に「次回復習まであと○日」「現在の間隔／EF」を小さく表示。
- ホームに**今後2週間の復習予定枚数（日別の棒グラフ）**＝負荷の見通し。
- 任意：項目ごとの**忘却曲線ミニグラフ**（復習ごとに減衰が緩くなる様子）。推定保持率は `R = exp(-Δ日 / S)`（S は interval に比例する安定度の目安）で近似してよい。学習科学の概念を可視化し動機づけにする。

## 厳守
- 既存スタック・規約に合わせる。スケジュール定数は1ファイルに集約。
- スケジュール状態は KV 同期（`last` でマージ）。オフラインでもローカルで判定・学習可能。
- 既存の取り込み・発音・録音・UIを壊さない。新ライブラリは理由を添えて確認。

## 進め方
1. 現状把握（既存SRS状態の有無・形式、復習導線）＋統合/移行計画を提示し承認を得る。
2. Sched 型と SM-2 更新関数（純関数・単体テスト付き）を実装。
3. 「今日の復習」自動キュー（due＋新規 `n` 順・1日上限）。
4. 3ボタン採点→Sched更新→次回due反映、再学習・苦手/リーチ。
5. 可視化（次回・予定枚数グラフ、任意で忘却曲線）。
6. 既存状態の移行と同期、受け入れ確認。
判断が要る箇所は勝手に決めず質問。

## 受け入れ基準
1. 起動時に「今日の復習」が自動生成され、due項目が古い順、新規が `n` 昇順で1日上限まで出る。
2. 忘れた→当日再出題＆間隔リセット、あいまい→間隔やや短縮、覚えてた→間隔が伸びる（1→6→×EF…）が確認できる。
3. EF が下限1.3で頭打ち、`reps`/`interval`/`due` が規則通り更新され、リロード・別端末同期後も保持。
4. `lapses>=2` で苦手、`>=8` でリーチ処理。数日空けても溜まったdueが古い順に出る。
5. 「次回復習まで○日」と今後の予定枚数が見える（忘却曲線グラフは任意）。
6. 既存機能（取り込み/発音/録音/セット学習）が従来どおり動く。
